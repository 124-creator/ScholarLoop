## 1. 模块目标

- **本模块解决什么**：把验证桩里只达到「可行性 go」的 A 主干，做成可对外声称「F1 明显优于基线」的**完整检索排序引擎**——在**全量 LitSearch** 上、同候选池下，用**显著性检验**证明 ScholarLoop 完整流程 F1 高于关键词 / BM25 / 神经 Embedding / 单轮 LLM。这是官方 70% 承重墙的落地件。
- **对冻结成功标准的贡献（映射 015 S1–S5）**：
  - **S1（F1 承重墙·70%）**：本模块的**主交付**——全量 LitSearch + 公开金标准 + 显著性优于基线。
  - **S2（效率·20%）**：每查询 API 次数 / Token / P50·P95 延时全程记录，纳入验收。
  - **S4（专家分·40%）**：创新=查询拆解 + 可解释综合排序；落地=可复现 + 真实学术 API 的连接器接口（本模块只留接口缝，离线 LitSearch 适配器先行）；泛化=全量评测 + 跨域可扩展设计。
  - **S5（合规）**：H5「0 虚构」强制；每条推荐带可解释理由，杜绝黑箱推荐红线。
  - **S3（结构化·10%）不在本模块**：由 M2（B-lite 证据矩阵）承担。

## 2. 范围

- **本轮必须做**：
  1. 语料 / 检索**接口层**（Repository 模式）+ 离线 **LitSearch 适配器**（复用验证桩已落盘的 64,183 篇 `corpus_clean`）。
  2. **查询理解 / 拆解**（LLM，temperature=0、seed 固定；复用 A-min 已验证拆解）。
  3. **混合召回**：BM25 + **神经稠密检索（040 §2 要求的 LSA→神经升级）**。
  4. **可解释综合排序**：分数融合 + 可选 LLM 重排；每条结果带「为何推荐」理由。
  5. **全量 LitSearch 评测**：P@10 / R@20 / F1，**对 BM25 做配对显著性检验**，并对四基线同候选池对比；效率与合规证据齐备。
- **明确不做**：B-lite 证据矩阵（M2）；C-lite（040 no-go，已砍）；在线多源连接器实际接入（仅设计接口 + 离线适配器）；Web 前端 / 后端服务；图谱 / 研究空白层；任何模型训练 / 微调；付费全文 / 未授权数据。
- **允许修改 / 新建的文件**：
  - 新建业务代码：`src/scholarloop/**`（多个小文件，高内聚低耦合）。
  - 新建测试：`tests/**`。
  - 新建评测运行器与产物：`src/scholarloop/eval/**`、`reports/m010/**`、`reports/m010/raw/llm/**`。
  - **不得**改动任何 FROZEN 件、`spike/**`（验证桩证据，只读复用其数据/方法）、本计划验收判据、`040`。

## 3. 现状与证据

- **已存在可复用**：
  - 验证桩 `spike/eval/run_spikes.py`：已验证的拆解 prompt、共享候选池协议、`metric_for_ranking`（P@10/R@20/F1）、`normalize_ranked_ids`（H5 池外即剔除）、LLM 调用封装（temperature=0/seed=42/空推理响应隔离）、密钥脱敏。**移植而非重写**。
  - 离线语料：`spike/raw/datasets/litsearch/corpus_clean/`（6 分片，64,183 篇）+ `spike/raw/datasets/hf/query/`（LitSearch 查询与金标准）。
  - LLM 端点：`LLM_BASE_URL`/`LLM_API_KEY`/`LLM_MODEL`（DeepSeek，`deepseek-v4-flash`，第三次初查实测有效；推理模型，`max_tokens` 须给足、读 `message.content`）。
- **已确认事实（040 / 005 §19）**：单轮 LLM 排序（R@20=0.27）**劣于** BM25（0.36）→ 排序必须以检索为主、LLM 仅辅助；A-min（BM25 下限 + 拆解 + LSA）仅 +0.03 R@20、无显著性 → 本模块的增益必须来自**神经 embedding + 更强融合 / 重排**并经显著性检验。
- **仍未知但可在执行中确认**：神经 embedding 升级后 A 主干能否在全量集上**显著**超过 BM25（核心风险，见 §11）；最佳融合权重；LLM 重排是否净正向（含效率代价）。

## 4. 调研结论

- **来源**：`015`（S1 金标准 + 阈值）、`035`（建造顺序 + 不可跨越边界）、`040` + `005`§19（spike 实测与前进路径）、LitSearch（EMNLP 2024, arXiv:2407.18940）。
- **可直接落地的结论**：
  1. 金标准 = LitSearch 官方相关性判定；指标 = F1 / R@20 / P@10；对比 = **同语料、同候选池**（候选池 = 各基线 Top-K 并集）。
  2. 神经稠密检索基线优先 `sentence-transformers/all-MiniLM-L6-v2`（小、CPU 可跑）为**地板**，允许升级到 `bge-small-en` / `e5-small-v2` 等开源更强模型（在 §10 预算内自行决定）。
  3. 「明显高于」必须量化为**配对显著性**（配对 bootstrap 95% CI 不含 0，或配对置换检验 p<0.05），不接受单点均值对比。
- **对本计划的影响**：T5（全量评测 + 显著性）是 make-or-break；若不显著须停机上报（见 §11），**不得**用挑样本 / 改判据制造胜利。

## 5. skills 选型与执行简报

- **选择的 skills**：无（本模块为工程实现 + 评测，执行方按本简报逐条执行即可；不把 skill 名抛给执行方）。
- **理由**：检索排序与评测是确定性工程任务，验证桩已给出可移植骨架；引入额外 skill 增加不确定性无收益。
- **转译后的普通执行步骤（工具无关，硬要求标【硬】 / 建议标【议】）**：
  1. 【硬】建 `src/scholarloop/` 包，按职责拆小文件（接口层 / 召回 / 排序 / 拆解 / 评测 / 配置），单文件 ≤400 行。
  2. 【硬】所有 LLM 调用 temperature=0、seed=42、原始响应落 `reports/m010/raw/llm/`、Token/延时入账；读 `message.content`。
  3. 【硬】候选池 = 各基线 Top-K 并集；所有系统在同池打分；推荐 ID 必须可在语料解析，池外/不可解析一律计不相关并记录（H5）。
  4. 【硬】全量 LitSearch 查询评测，输出逐查询 + 聚合 + 配对显著性。
  5. 【议】融合权重与是否启用 LLM 重排，可在 §10 预算内调参，但调参依据与最终配置须落盘。

## 6. 任务拆解

- [ ] **T0（不确定度：低）凭证自加载配置模块**
  - 背景：执行方运行于 IDE 插件 / 桌面 App，外部终端 `$env:` 不进其进程（rev2、M010 三次停机均因此）。
  - 动作：建 `src/scholarloop/config.py`，import 时若 env 缺失则从 `secrets/llm.env.local` 加载进 `os.environ`（运行时读文件）；所有 LLM 代码先 import 它；端点预检在 import 之后。
  - 产出：`src/scholarloop/config.py`。
  - 完成判据：`python -c "import sys; sys.path.insert(0,'src'); import scholarloop.config as c; print(c.PRESENT)"` 三项均 True。
  - 红线：该凭证文件只读、不创建/修改/回显/复制；env 在写代码前为空**不构成停机理由**，仅当加载后仍缺或端点真实拒绝才停机。

- [ ] **T1（不确定度：低）语料 / 检索接口层 + LitSearch 适配器**
  - 输入：`spike/raw/datasets/litsearch/`、`.../hf/query/`。
  - 动作：定义 `CorpusRepository` 接口（`search/get/iter`），实现 `LitSearchCorpus`；预留 `OpenAlex/SemanticScholar/Crossref/Arxiv` 连接器接口签名（**不实现在线调用**）。
  - 产出：`src/scholarloop/corpus/`；加载 64,183 篇 + 查询/金标准的单元测试。
  - 完成判据：加载篇数 = 64,183，金标准查询全部命中语料（缺失即报错）。

- [ ] **T2（不确定度：中）神经稠密检索（LSA→神经升级）**
  - 动作：用开源句向量模型对语料编码、建向量索引（精确或 FAISS/numpy），查询侧同模型编码、余弦召回 Top-K；编码确定性（固定模型版本 + 无随机 dropout）。
  - 产出：`src/scholarloop/retrieval/dense.py` + BM25 模块 `bm25.py`。
  - 完成判据：两次编码/检索同输入同输出；记录模型名+版本入证据；neural≠LSA（断言）。
  - 【停机】若需 GPU 才能在可接受时间内编码全量语料，或需 >2GB 模型下载受阻 → 停机上报（见 §11），不擅自换重依赖。

- [ ] **T3（不确定度：中）查询理解 / 拆解**
  - 动作：移植 A-min 拆解（子查询 + 相关性判据），temperature=0、seed=42、原始响应落盘。
  - 产出：`src/scholarloop/query/decompose.py`。
  - 完成判据：每查询产出 ≤4 子查询 + 判据；空/无效推理响应隔离不复用。

- [ ] **T4（不确定度：中）混合召回 + 可解释综合排序**
  - 动作：对（原查询 + 子查询）各跑 BM25 + 神经召回，融合到共享候选池；分数融合（BM25 + dense + 子查询证据）排序；可选 LLM 重排 Top-N；**每条结果产出理由**（命中的子查询 / 判据 / 信号）。
  - 产出：`src/scholarloop/rank/fusion.py`、`rank/rerank.py`、`pipeline.py`（串起 T1–T4）。
  - 完成判据：输出含 `corpusid + score + reason`；reason 非空；无黑箱（每条可追溯到信号）。

- [ ] **T5（不确定度：高）全量 LitSearch 评测 + 显著性检验** ← make-or-break
  - 动作：在全量查询集上对 5 系统（关键词 / BM25 / 神经 Embedding / 单轮 LLM / ScholarLoop-A）同候选池算 P@10/R@20/F1；对 ScholarLoop-A vs BM25 做**配对 bootstrap（≥10000 重采样）或配对置换检验**；记录效率（API 次数 / Token / P50·P95 延时）。
  - 产出：`reports/m010/results.json`+`.csv`、`reports/m010/significance.json`、`reports/m010/A主干评测报告.md`。
  - 完成判据：见 §8 验收。
  - 【停机】若 ScholarLoop-A 未**显著**优于 BM25 → 停机上报，**不得**挑样本 / 改判据 / 关基线制造胜利。

- [ ] **T6（不确定度：低）合规 / 可复现 / 复盘**
  - 动作：全程 grep 无密钥落盘；两次端到端跑结果一致；写复盘。
  - 产出：`../retrospectives/010-A主干检索排序引擎-复盘.md`。
  - 完成判据：reproducible=True；零密钥；复盘含失败模式与诚实结论。

## 11. 风险、边界与回滚

- **核心风险（T5）**：验证桩仅显示 A-min 比 BM25 +0.03 R@20 且不显著。本模块押注「神经 embedding + 更强融合 / 重排」把差距做到显著。**若失败**，这是真实的方法信号——停机上报，由总指挥 / 人类决定：加强重排、引入更强 embedding、或回到路线层重议；**绝不**伪造胜利。
- **边界（035 §4 继承）**：不做通用聊天 / 关键词站 / 纯标题列表 / 黑箱推荐 / 伪造引用 / GPU 大训练 / 未授权付费全文。
- **回滚**：本模块代码独立于 `spike/**` 与 FROZEN 件；失败仅废弃 `src/scholarloop/**` 与 `reports/m010/**`，不污染验证桩证据。
