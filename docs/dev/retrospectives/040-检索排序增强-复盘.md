## 1. 本轮做了什么

- 新增 A-v2 独立路径，不回改 M010/M020/M030：
  - `src/scholarloop/retrieval/dense_v2.py`
  - `src/scholarloop/rank/rerank.py`
  - `src/scholarloop/rank/fusion_v2.py`
  - `src/scholarloop/eval/run_full_v2.py`
- 神经嵌入从 M010 `all-MiniLM-L6-v2` 升级为开源离线 `BAAI/bge-small-en-v1.5`。
- Cross-encoder 使用开源离线 `cross-encoder/ms-marco-MiniLM-L-6-v2`，对候选池 Top-50 精排并做开/关消融。
- 启用 M010 已落盘 query decomposition 作为子查询信号：`sub_bm25/sub_dense_v2` 在全量上进入融合网格。
- 防过拟合：按 `sha256(query_id) % 10` 做 train/holdout/test（0-5 train，6-7 holdout，8-9 test）；融合权重只在 train 上选择。

## 2. 模型/资源决策

- 初始尝试 `BAAI/bge-base-en-v1.5`，CPU 全量编码在工具 3 小时上限内未生成可用 dense 缓存。
- 按 040 §10 的模型有界决策，改用同属 BGE 系列、开源、可离线、下载量远低于 2GB 的 `BAAI/bge-small-en-v1.5`；并将 dense 编码改为分块缓存，避免长跑失败无中间产物。
- 未使用 LLM 调用；A-v2 token=0，API calls/query=0。

## 5. 已知风险 / 给初查的重点

- Cross-encoder 净增益为正但未显著：A-v2 vs no-rerank ΔF1=0.003137，95% CI [-0.000959, 0.007106]。最终 A-v2 仍通过三条承重墙；若后续追求简化，可评估是否保留 cross-encoder 作为 NDCG/排序质量增强而非 F1 承重件。
- 当前最终模型用 `bge-small` 而非 `bge-base`，原因是 CPU 首跑可交付性。若比赛环境有稳定 GPU，可作为后续优化尝试，但本轮不把 GPU 作为必要条件。
- A-v1 与 single-LLM 作为 M010 冻结对照读取，不重新触发 LLM；这是为了遵守 M010 冻结与 0 新 LLM 调用边界。
