## 0. 冻结边界复述（不可改动，仅供评审锚定）

本文件**不重新定义** `010-问题定义.md` 中已冻结的目标、成功标准与硬约束，仅复述要点用于判断三条路线是否达标。

**冻结目标**：面向研究生复杂科研问题的可信论文搜索、证据链组织与研究空白发现智能体（ScholarLoop），优先 Web 形态，输出可用于开题、综述、选题、创新点发现的结构化科研证据链。

**五条成功标准（要点）**：
1. 功能完整：≥6 个核心模块（输入 / 查询拆解 / 多源检索 / 过滤排序 / 证据矩阵与归因 / 研究空白与选题建议）；Web 可完整演示；≥3 个完整案例。
2. 检索效果：≥50 题复杂查询评测集；与普通关键词、BM25、Embedding Search、单轮 LLM、完整流程对比；**≥2 个核心指标优于基线**；人工抽检合理率 ≥85%；不得虚构论文 / DOI / 作者 / 期刊。
3. 证据链完整：每篇论文输出 10 项字段（标题、作者年份、来源/DOI、推荐理由、支持问题、方法、数据、结论、局限、关联强度）；显式区分“已有证据支持 / 证据不足 / 存在争议 / 需人工核验”。
4. 创新性：不能停留在普通 RAG 或搜索网页，须体现查询拆解、多轮检索与自我修正、综合排序、证据矩阵化、研究空白发现、可复现评测与基线/消融。
5. 比赛交付：Web Demo、技术报告、架构图、算法流程图、评测集说明、对比表、消融表、≥3 案例、演示视频、README、核心代码、风险与局限说明。

**硬约束（要点）**：个人参赛、工程可控；优先公开数据库 / 公开 API / 本地可复现库；不依赖昂贵 GPU、不可公开数据、不可复现闭源流程；不用未授权付费全文；过程与结果须可保存、可复查、可复现；不伪造引用；不做通用聊天机器人 / 普通关键词搜索站 / 仅标题列表 / 无评测展示 / 黑箱推荐；碳价预测不得作为唯一核心场景。

**三条候选路线**：
* **A — 评测驱动的可解释检索排序**：以可解释评分、可复现评测、稳定输出为中心。
* **B — 证据图谱驱动的研究空白发现**：以“问题—论文—方法—数据—结论—局限”结构化图谱为中心。
* **C — 审稿式多智能体自我修正**：以多轮规划、检索、质检、补检为中心。

## 1. 评审口径

**评审目标**：不是证明所有路线都不行，而是找出**真正致命**的问题，并明确各路线在补齐关键缺口后能否继续。

**分级定义**：
* **must-fix**：真正阻断推进到“验证桩”的问题。不补齐，进入验证桩也验证不了任何东西。每条路线**路线特有 must-fix ≤5 条**；共性 must-fix 见 §2，对三条路线同时生效。
* **重要**：显著影响最终质量与说服力，但**不阻断最小验证**，可在验证桩之后处理。
* **可选**：改了更好，当前可不做。

**绿灯结论定义**：
* **GREEN**：可直接进入验证桩。
* **CONDITIONAL GREEN**：补齐指定 must-fix（含共性）后可进入验证桩。
* **RED**：不建议继续。

**一个贯穿全篇的判断**：这个项目的承重墙不是“哪种检索架构更花哨”，而是**“你凭什么证明自己比基线好、而且这个证明可信”**。成功标准 2 同时要求“≥2 指标优于基线”和“人工抽检合理率 ≥85%”，二者都依赖一套**可信的评测集和实验协议**。三条路线在架构上各有取向，但**都把这套评测协议当成了已解决的前提**——这是它们共同的、也是最大的盲区，因此单列为 §2。

## 2. 共性致命问题（X1–X3，对 A / B / C 同时生效，均为 must-fix）

这三条不是某条路线的差异化短板，而是三条路线**共享的拦路石**。为避免在每条路线下重复三遍，统一在此列出；§3–§5 各路线 must-fix 会引用 `X1–X3`。

**X1. 金标准 / 相关性判定协议缺失，且“自建自评”构成致命可信性漏洞。**
要算 Precision@10、NDCG、Recall@20，必须为 50 个复杂查询提供相关性判定（ground truth）。文档从未说明：标签由谁产生、候选池如何构造、判定规则是什么。更要命的是——设计系统的人同时标注金标准，评测是**自证循环**，任何一个有经验的技术评审都会一句话击穿：“你赢的是你自己定义的标准。”
*前进路径*：写明 (a) 查询来源与领域分布；(b) 候选池构造采用**混合池化**（pooling，把各系统 Top-K 合并后统一判定，避免只判自家结果）；(c) 成文的判定 rubric；(d) **至少引入一个第二判定来源**（第二位标注者抽样 + 报告一致性，或外部锚点如已有综述/被引数据），并显式说明利益冲突如何缓解。这是“能不能拿出可信指标”的前提，不补齐则成功标准 2 无法成立。

**X2. 基线对比协议未定义，极易“赢一个稻草人”或“被判作弊”。**
“在 ≥2 指标上优于基线”——但基线（普通关键词 / BM25 / Embedding / 单轮 LLM）跑在**什么语料 / 什么候选池**上？若 ScholarLoop 与基线不共享同一候选池，就是苹果比橘子。尤其“单轮 LLM 检索”这个基线：如果允许它虚构论文、而 ScholarLoop 不允许，对比直接被判不公平。
*前进路径*：固定一个受控实验协议——同一公开语料 / 同一候选召回上限、基线 LLM 的虚构如何计入或剔除写清楚、所有系统在同一评测集同一判定下评分。否则“优于基线”不可信。

**X3. 数据来源始终未点名，许可 / 字段覆盖 / 频率限制未核验。**
三条路线都假设“公开元数据 + 摘要 + 开放全文片段”够用，却**没有点名任何一个具体来源**，也没核验它能否在不用付费全文的前提下支撑 10 项证据字段。验证桩连检索都没法接。
*前进路径*：点名候选公开来源（如 OpenAlex、Semantic Scholar、Crossref、arXiv 等——**具体许可、字段覆盖与速率限制须自行核验，本文件不替你断言**），并用小样本确认摘要/开放片段能否支撑“方法 / 数据 / 结论 / 局限”的抽取与归因。这是硬约束 4 的直接落地，也是验证桩的输入前提。

> X1–X3 是“真正拦路石”，不是找茬：它们不解决，任何一条路线的核心卖点（“我比基线好、证据可信”）都立不住。

## 3. 路线 A 评审：评测驱动的可解释检索排序

**一句话定位**：三条路线里**最可交付、最可控、最不容易翻车**的一条；天然产出成功标准 2/5 所需的评测/基线/消融物料；幻觉与复现风险最低。它的问题不在“能不能跑”，而在“够不够创新”。

### 3.1 must-fix（路线特有，另需满足共性 X1–X3）

1. **先做“数据充分性”验证桩，再投入。** 取 ~5 个复杂查询，在已点名来源上确认：仅靠摘要/开放片段能否支撑 10 项证据字段的真实归因。若不能，A 的“证据矩阵”是空壳，成功标准 3 直接崩。——这正是 A 在 2.7 自报的最危险假设，是对的 spike 目标。
2. **把“证据归因准确率”操作化为可测指标。** 它被 2.8 列为 A 争取的 ≥2 个获胜指标之一，却从未定义“归因正确”如何判定、谁判、样本量多少。未定义 = 不可核验 = 不能用来声称优于基线。
3. **在路线层就锁定一个“创新性差异化”决策。** 纯排序流水线高度面临成功标准 4 的“不能停留在普通 RAG”红线（A 自己在 2.4.1/2.4.2 承认创新感与空白发现偏弱）。必须**现在**就决定至少引入一个差异化件——证据矩阵层，或一个“自我修正”消融臂——否则会推进进一个“注定过不了创新关”的验证桩。

> 说明：A 的“最小流水线能跑”不难，难的是“跑出来的东西过得了成功标准 4”。上述 3 条加 X1–X3 才是 A 进入验证桩的完整阻断集。

### 3.2 重要（影响质量，不阻断最小验证）

* **综合排序权重的来路。** 相关性/权威性/新颖性/证据强度/多样性的权重若是手调，就是“调到 demo 好看”，A 自己在 2.4.4 承认这是主观打分风险。应**预注册权重**或**用留出集学习权重**，并补一张权重消融表。这是 A 的“核心资产”，做实了才撑得起“可解释”。
* **研究空白发现的强度。** A 的空白发现偏规则化总结，冲击力弱于 B。成功标准 4 明确要求空白发现，最小版本可上，但需想清楚“规则化空白”如何不被评委看作凑数。
* **创新叙事。** 需要一句话把 A 讲成“复杂科研问题不是直接搜，而是拆解—召回—排序—归因—评测的可信闭环”，否则容易被一句“增强版 RAG”带走。

### 3.3 可选

* 证据矩阵的展示美化；多样性指标（如 MMR）的公式细化；API 结果缓存层以加速 demo。

### 3.4 最危险假设是否选对

**方向对，但没选中最致命的那个。** A 在 2.7 选了“公开元数据/摘要/开放片段足以支撑可信归因与排序优势”——这是一个真实、且值得做 spike 的高价值假设。但 A 真正最危险的假设是 **“一个人自建自评的 50 题评测集，会被评委当作‘优于基线’的可信证据”**（即 X1）。A 看到了数据可用性风险，却低估了**测量可信性**风险——后者一旦不成立，A 的全部指标优势都失效。

### 3.5 绿灯结论

**CONDITIONAL GREEN（三条路线中风险最低）。** 补齐 X1–X3 + 3.1 的三条后可进入验证桩。但须显式标注：A 在成功标准 1/2/5 上达标可达，**在成功标准 4（创新性）上处于风险区**，除非按 3.1.3 落实差异化件。A 的失败模式是“优雅降级”——即便创新不够，仍是一个能跑、能测的系统，不会归零。

## 4. 路线 B 评审：证据图谱驱动的研究空白发现

**一句话定位**：**幻灯片上最像冠军、个人交付最可能翻车**的一条。创新与冲击力最高，但图谱抽取质量是长杆，误差会沿“抽取→矩阵→图谱→空白判断”逐级放大。

### 4.1 must-fix（路线特有，另需满足共性 X1–X3）

1. **为图谱抽取设定准确率指标 + 底线 + 验证方法。** 方法/数据/结论/局限若抽取准确率只有六七成，“研究空白”大多是抽取假象而非真实空白（B 在 3.4.1 自承误差放大）。建图层之前就要把抽取准确率测出来、设底线。
2. **让“研究空白”可证伪。** 图谱里的“空缺/弱连接”极易制造、也极易无意义——缺一条边通常意味着“你没检到/没抽到”，而非“学界没研究过”。评委必问：“你如何区分真实空白与检索/抽取漏洞？”没有可辩护的回答 + 专家验证（被标记的空白确实真实且非平凡），B 的头号差异化件就是不可证伪的，属致命。
3. **先验证跨案例通用证据结构，再投入。** 用 2–3 个跨领域小图谱确认一套 schema 能否同时覆盖碳价预测、AI 科研智能体与跨学科案例（B 在 3.7 自报的最危险假设，选得对）。若需频繁改 schema，评测集/矩阵/图谱/空白判断无法统一，“通用智能体”定位也随之削弱。
4. **写下硬性范围上限。** B 在 A 的全部工作量之外再叠加 schema 设计、图谱抽取、图谱存储、图谱可视化、空白评价 rubric——对个人参赛（硬约束 1）是最高范围风险（B 在 3.4.4 自承范围膨胀）。必须成文锁定：每案例节点上限、冻结 schema、可视化定为**静态只读**而非交互物理图，否则会吃光时间预算、饿死评测/基线/案例这些必交付物料。

### 4.2 重要

* **可视化沦为炫技**：冻结 out-of-scope 第 10 条明确反对“把页面视觉置于证据链之上”，图谱页极易踩线。
* **现场图谱生成稳定性**（B 的 3.4.5）：demo 案例应预计算，不在答辩时现算。
* **基线可比性更难**：图谱系统 vs BM25 更像苹果比橘子；必须让**检索层仍可比**，把图谱定位为检索后的“组织层”，否则对比无意义。

### 4.3 可选

* 争议边聚类、空白热力图、交互式图谱探索。

### 4.4 最危险假设是否选对

**选对了一个关键假设，但漏掉了同等或更致命的那个。** B 在 3.7 选“跨案例通用 schema 能稳定覆盖”——这是真实的结构性关键，且是好的 spike 目标。但它属于**可行性**风险；B 漏掉的是**有效性**风险：**“图谱空缺 = 真实可发表的研究空白”这一推断本身可能不成立**（见 4.1.2）。对一个以创新可信度评分的比赛，后者更致命：schema 不通用最多让你做得累，空白推断无效则让你的头号卖点根本不存在。

### 4.5 绿灯结论

**CONDITIONAL GREEN（高天花板、高方差）。** 补齐 X1–X3 + 4.1 四条**且范围上限真正落地**后可进入验证桩。但需诚实标注：纯 B 形态下，完整“图谱 + 空白评价”对个人参赛在时限内交付的把握偏低。**强烈建议把 B 降级为 A 之上的一个差异化层**（见 §7），而非主路线——那样能拿到 B 八成的创新价值，避开 B 的交付塌方风险。

## 5. 路线 C 评审：审稿式多智能体自我修正

**一句话定位**：**智能体叙事最强、与冻结硬约束冲突最多**的一条。它最贴“智能体”定位、最利于消融，但与“可复现”（成功标准 2 + 硬约束 5）、“不可复现闭源流程”（硬约束 3）、“个人可控”（硬约束 1）正面冲突。

### 5.1 must-fix（路线特有，另需满足共性 X1–X3）

1. **复现性策略，直接对冲冻结约束。** 多轮 LLM 调用必然有运行间方差（C 在 4.4.1 自承）：同一查询重跑给出不同推荐，则 (a) Precision@10 不可复现，(b) 评委现场重跑可能拿到更差结果。必须：温度 0、可固定处固定随机种子、**评测与 demo 使用冻结/缓存的运行产物**、并报告 N 次运行的 mean±std。不补齐则违反冻结的可复现要求。
2. **尽早做 go/no-go 消融。** 单轮 vs 多轮 vs +质检，必须在 ≥2 指标上展示**真实且可复现的提升**；若没有提升，C 就只是“加了步骤的、又慢又贵的 RAG”（C 在 4.4.3 自承“套壳智能体”风险），应当放弃。这是 C 存在的理由，必须是**第一优先级早期实验**，不是最后一周补的。
3. **质检改为确定性校验，而非 LLM 自评。** 冻结约束禁止虚构 DOI/作者，要求真实来源核验。一个“质检 agent”说“这条引用没问题”**不是核验**（C 在 4.4.4 自承质检 ≠ 真实正确）。质检必须落在确定性检查上：DOI 能在 Crossref/OpenAlex 解析、作者/年份/期刊与 API 记录匹配。否则“自我修正降低幻觉”这一头号卖点没有依据，会被评委当场抓住。
4. **延迟/成本预算 + 预缓存演示路径。** 多角色 × 多轮 × 多源的现场调用，在限时答辩里运营风险高（C 在 4.4.2 自承）。必须给出延迟/成本预算，并准备预缓存演示路径，使答辩不依赖现场多轮调用按时完成（关系到成功标准 5 的可演示性）。
5. **限角色、限轮次、早冻结编排。** 角色 ≤3–4、补检轮次 ≤2、尽早冻结编排，保护评测/基线/案例这些必交付物料（硬约束 1；C 在 4.4.5 自承流程复杂度会压缩核心实验时间）。

### 5.2 重要

* **过程日志需压成评委易读的主线**（C 的 4.4.5），而非一墙日志。
* **“单轮 LLM 检索”基线与 C 的第一轮高度重叠**：必须把基线定义成干净、独立的系统，否则消融含混。

### 5.3 可选

* 动画化过程时间线、自适应轮次。

### 5.4 最危险假设是否选对

**三条里选得最准。** C 在 4.7 选“多轮自我修正能在不破坏可复现性与演示稳定性的前提下显著提升 ≥2 指标”——它恰好把两个存亡风险**打包**进了一个假设。唯一改进：**拆成两个可独立检验的门**——(a) 提升是否存在（消融），(b) 提升是否可复现（方差报告）——因为这两者会独立失败。

### 5.5 绿灯结论

**CONDITIONAL GREEN（带闸门，最有条件的一条）。** 补齐 X1–X3 + 5.1 五条**且 go/no-go 消融（5.1.2）通过**后可进入验证桩。**若早期 spike 显示没有可复现的提升，C 直接翻为 RED。** C 的失败模式是“灾难性”而非优雅降级：拿不到提升时，剩下的是一个又慢、又贵、又难复现的智能体外壳。

## 7. 推荐路线与理由

**推荐：以 A 为主干（spine），并强制叠加一个轻量差异化件——一个有界的“证据矩阵层”（取自 B）+ 一个“自我修正”消融臂（取自 C）。**

这不是发明第四条路线，而是 **“A，经过最小增强”**；它不改变冻结目标与路线集，只是回答“先建什么、用什么补创新”。

**理由（用四个反方视角各说一句）**：
* **Hiring Manager（能不能交付）**：A 是唯一一条让个人参赛者在时限内**几乎确定**交齐成功标准 2/5 全套物料的路线。B/C 的纯形态都有现实的交付塌方风险。
* **技术评审（实验严不严）**：A 的评测/基线/消融是原生产物，X1–X3 一旦补齐，A 的指标最经得起复现质询。B 多出两类需 rubric 的指标，C 与复现约束正面冲突——都更难过审。
* **项目反方（第三周死在哪）**：纯 A 的死法是“这不就是 RAG 吗”（成功标准 4）；纯 B 死在图谱抽取与空白证伪；纯 C 死在延迟/方差。把 B 的证据矩阵层与 C 的一个消融臂叠到 A 上，**正好补上 A 唯一的硬伤（创新）**，且二者都是 A 流水线的自然下游（必须先检索排序，才谈得上矩阵化与“补检是否有用”）。
* **HR（这份候选画像可信吗）**：A+轻增强讲出的故事是“稳、可测、且有一个真实的差异化点”，比“图谱很炫但没做完”或“智能体很酷但重跑结果会变”都更像能拿一等奖的成熟判断。

**建造顺序**：A 流水线 → 证据矩阵层（B-lite）→ 一个自我修正消融臂（C-lite）→（仅当时间充裕且 4.1.3 的 schema spike 通过）再考虑图谱/空白层。

**若收敛阶段坚持要单一纯路线**：选 **A**。它是唯一一条 must-fix **全部可由个人在时间预算内解决**、且失败优雅降级的路线。

## 8. 真正拦路石 vs 评审尽职找茬

按评审要求，明确区分二者，避免把找茬当成阻断。

**真正拦路石（不解决就达不到冻结成功标准）**：
* X1 评测金标准/相关性判定协议缺失 + 自建自评利益冲突（全路线）。
* X2 基线对比协议未定义、苹果比橘子（全路线）。
* X3 数据来源未点名、许可/字段/频率未核验（全路线）。
* B：图谱抽取准确率无底线、无评测；“图谱空缺 = 真实研究空白”不可证伪。
* C：非确定性违反冻结复现约束；LLM 质检 ≠ 真实核验；自我修正提升未经证实。

**评审尽职找茬（值得记一笔，但不阻断最小验证）**：
* 具体可视化美化程度、配色、动画。
* 选哪一个 embedding 模型。
* 缓存/性能微优化。
* 多样性指标用 MMR 还是别的公式。
* 重排序具体算法选型。
* 过程日志的措辞细节。
> 这些是“为了显得尽职”会提、但不影响最小验证能否通过的问题。收敛阶段不应被它们拖入返工。
