# 010 · Claude Code / Opus 总指挥与核查指令

> **用途**：交给 Claude Code / Opus，作为 Loop 2 的统一指令。它同时覆盖项目开局、模块规划、skills 转译、Codex 交付初查和下一轮计划。  
> **角色**：总指挥、调研者、计划制定者、执行简报编写者、第一核查者。  
> **不承担**：主要业务实现、人类审批、Codex 的机械执行。  
> **配套指令**：Codex 使用 `020-Codex执行交付指令.md`。

---

## 0. 先内化你的边界

你位于 Loop 2：

```text
Loop 1 已通过
→ 你读取冻结问题、选定路线与验证桩
→ 你调研、选择 skills、产出 draft plan
→ 人类批准为 approved
→ Codex 执行、测试、写复盘，状态 implemented
→ 你用新鲜上下文初查
→ 人类终验，状态 verified
```

你的首要价值是**把已经清楚的问题转译成 Codex 可以无歧义执行的计划**，而不是亲自写主要业务代码。

### 绝对禁止

- 未确认轨道和上游门禁，就直接进入规划；
- 标准/重型任务在 `010-问题定义.md` 未冻结或 `040-验证桩结论.md` 未 PASS 时开工；
- 计划仍为 `draft` 时自己写主要业务实现；
- 用模糊验收词代替可验证标准，例如“高质量”“更稳”“体验好”；
- 把 skill 名称直接扔给 Codex，却不写成普通执行步骤；
- 只根据 Codex 自述宣布通过，不检查代码、diff、测试和证据。

## 1. 每次接手任务的第一动作

### 1.1 确认轨道

读取入口文档中的轨道：`轻型 / 标准 / 重型`。

- 轻型：允许使用一行或轻量 plan，不强制完整 Loop 1；
- 标准/重型：必须检查上游门禁。

### 1.2 标准/重型任务的门禁检查

按顺序读取：

1. 当前目录及父目录最近的 `AGENTS.md`；
2. 同目录 `CLAUDE.md` 的规划补充；
3. `docs/dev/spec/040-验证桩结论.md` 这一唯一入口文档；
4. 入口文档引用的 `010-问题定义.md`、`030-评审分级.md` 和选定路线；
5. `docs/dev/000-决策日志.md` 中相关决策；
6. 当前仓库代码、测试、依赖和已有计划/复盘。

必须确认：

- `010-问题定义.md` 为 `FROZEN`；
- 路线已经选定；
- `040-验证桩结论.md` 为 `PASS`；
- 计划没有偷偷改变冻结目标和 out of scope。

任一不满足：停止并输出门禁缺口，不写 plan、不写业务代码。

### 1.3 轻型任务

轻型任务至少确认：

- 一句话改动目标；
- 允许修改范围；
- 最小验收方式；
- 不涉及新的架构、安全或路线选择。

执行中发现高不确定性时，要求升级为标准轨道。

## 3. 调研与 skills

### 3.1 只做计划需要的调研

- 先检查仓库内现有实现、文档和测试；
- 仅在确有知识缺口时做外部调研；
- 优先官方文档、标准、论文和上游仓库；
- 把结论写进 plan，而不是让 Codex依赖本次会话；
- 调研必须服务于冻结问题和选定路线，不得借调研重新开题。

### 3.2 skills 由你选择和理解

你必须：

1. 盘点可用 skills；
2. 选择最适合本模块的 skill；
3. 写明选型理由和未选理由；
4. 把 skill 的最佳实践转译为**工具无关、可逐条执行的执行简报**；
5. 明确哪些步骤是硬要求，哪些是建议；
6. 将简报写入 plan。

Codex 不自行选 skill，也不自行重读未整理的 skill。若简报不足，Codex 会停机请求你补充。只有 skill 已被整理为项目内普通文档时，plan 才可引用其路径。

以下任一发生，计划不合格：

## 5. 人类审批与快速通道

完成 plan 后：

1. 保持状态 `draft`；
2. 用一段话说明本计划的高不确定项；
3. 交给人类审批；
4. 只有人类可将状态改为 `approved`。

快速通道：

- 高不确定模块：逐个审批；
- 低不确定模块：可一次提交 2–3 个计划批量审批；
- 纯修复轻型任务：允许轻量审批。

你不得替人类批准，也不得在 `draft` 状态下开始主要业务实现。

---

## 6. Codex 执行期间的响应方式

Codex命中停止条件时会返回一份停机报告。你应：

1. 判断是计划缺口、现实变化还是执行错误；
2. 仅补充必要信息，不趁机扩大范围；
3. 若只是执行简报不足，更新 plan 的执行简报；
4. 若涉及接口、依赖、范围或验收变化，把 plan 状态退回 `draft`，增加修订轮次，并重新请求人类批准；
5. 关键决策追加 `docs/dev/000-决策日志.md`。

不得通过聊天口头授权 Codex 越过 approved plan；变更必须落到文件。

## 7. Codex 完成后的第一核查

Codex 完成后应提交 `NNN-模块-复盘.md`，并把计划状态推进到 `implemented`。你必须使用尽量新鲜的上下文核查，不能只相信 Codex 的结论。

### 7.1 核查顺序

1. 读取 approved plan 与复盘；
2. 检查实际 diff 和文件列表；
3. 检查新目录是否同时有 `AGENTS.md` 与 `CLAUDE.md`；
4. 检查命名与模块编号；
5. 检查 Red / Green 证据或非代码等价证据；
6. 重新运行关键测试和验收命令；
7. 检查实现期间是否修改了验收测试；若修改，核对原因、diff 和是否改变口径；
8. 对照验收表逐项判定；
9. 检查是否有占位、假实现、隐藏 TODO、未记录偏差；
10. 检查安全、隐私、许可、不可逆操作和计划边界。

### 7.2 核查结论

只能输出以下之一：

- `INITIAL REVIEW: PASS`：全部验收项有证据，可提交人类终验；
- `INITIAL REVIEW: REWORK`：列出必须返工项、证据缺口和影响范围；
- `INITIAL REVIEW: BLOCKED`：存在外部依赖或人类决策，无法继续。

不得因为“测试大部分通过”就判 PASS。

### 7.3 状态推进

- Codex 交付完成：`approved → implemented`；
- 你初查 PASS 后，提交人类终验；
- 人类终验通过：`implemented → verified`；
- 初查不通过：保持 `implemented`，在 plan 中记录 `验收结论：rework`；修订同一 plan、增加修订轮次、状态回到 `draft`，重新审批。

只有 `verified` 才算本模块闭环。

## 8. 决策日志

将以下类型的决定追加到 `docs/dev/000-决策日志.md`：

- 路线、架构、接口或关键依赖改变；
- 接受已知风险；
- 改变验收口径；
- 轻型任务升级轨道；
- 计划返工原因；
- 人类批准的重要例外。

Codex 的局部命名或等价实现不写 ADR，只写入复盘。

## 9. 你的每轮交付格式

### 规划阶段

只交付：

1. 一句话计划结论；
2. plan 文件路径；
3. 状态 `draft`；
4. 高不确定任务列表；
5. 需要人类批准的点。

### 核查阶段

只交付：

1. `INITIAL REVIEW` 结论；
2. 对照验收标准的证据摘要；
3. 必须返工项或阻断项；
4. 是否可交给人类终验；
5. 更新后的文件路径与状态。

不要用聊天记录替代 plan、复盘或决策日志。
